# Task 1239 Doctrinal Review Plan

## Purpose

Create a bounded doctrinal review path for Narada proper `packages/` and the repo-local `.narada` Site substrate.

This plan is an execution guide, not a review finding. Review conclusions belong to the follow-up tasks created from this plan.

## Target Locus

- Authority root: `D:\code\narada`
- Site root: `D:\code\narada\.narada`
- Authority basis: `.narada/site.json` records Windows-native Narada proper authority for repo, package, task, inbox, admission, capability, and `.narada` Site substrate mutations through governed tasks and evidence.

## Doctrine Lenses

- Authority-homogeneous zones and governed crossings.
- Intelligence-Authority Separation: description, recommendation, claimed identity, and authority-bearing mutation remain distinct.
- Plural Embodiment, Singular Authority: Windows-native Narada proper is the current authority root; WSL, external Sites, and carrier projections are embodiments or evidence, not implicit authority.
- Inhabited Evolution: package and Site machinery must be earned from used operations, with unearned surfaces recorded as candidates or residuals.
- Constructive universalization by re-instantiation: reusable packages must support future Sites through contracts/templates, not source-Site state copying.
- Canonical Mutation Evidence: SQLite/runtime substrate needs Git-visible evidence for governed mutation or reconciliation.
- Governed Crossing: external handoffs, package adoption, Site setup, MCP exposure, and operator-surface delivery must preserve arrival/admission/execution/truth separation.

## Review Task Split

1. `packages/` doctrinal review
   - Scope: package boundaries, package names, source/docs/tests, CLI surfaces that expose package behavior, and package-local README/ADR posture where present.
   - Focus: reusable package purity, descriptor/adapter boundaries, Windows-native posture, no source-Site state import, no package-owned live authority unless explicitly admitted, CLI exposure clarity.
   - Output: `.narada/audit/task-1240-packages-doctrinal-review.md`.

2. `.narada` Site doctrinal review
   - Scope: `.narada/site.json`, admission records, capabilities, tasks, surfaces, crew descriptors, checkpoints, local state descriptors, audit records, and policy fragments.
   - Focus: Site authority declaration, capability lifecycle state accuracy, admitted/non-admitted mutation boundaries, direct substrate shortcut posture, evidence chain, runtime state versus Git-visible evidence.
   - Output: `.narada/audit/task-1241-site-doctrinal-review.md`.

## Evidence Shape

Each execution review should record:

- review scope and exact sampling method;
- grounded doctrine lenses used;
- findings with severity and file/path references;
- explicit non-findings when posture is coherent;
- residual risks and recommended follow-up tasks only when needed;
- verification commands and output summaries.

## Non-Goals

- Do not import narada-andrey, CPY, PC-locus, or other source Site runtime state.
- Do not mutate live DBs, MCP registrations, operator-surface runtime, Windows profiles, shortcuts, credentials, or PC state.
- Do not perform broad package refactors while reviewing.
- Do not claim complete formal proof of all `packages/` or `.narada` content; this is a bounded doctrinal review pass.
