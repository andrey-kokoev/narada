# Live Graph Proof Рђћ Draft-to-Confirmed Saturation

> The canonical acceptance proof for Narada's full live Microsoft Graph loop: draft creation, approval/send submission, and reconciliation after real send.

This document defines the **live-proof contract** for the mailbox vertical. It specifies what must be observable at each stage of the pipeline, the canonical commands to inspect state, and the boundary between public-repo artifacts and private operational evidence.

For the fixture-backed proof (mechanical pipeline correctness without live credentials), see [`docs/first-operation-proof.md`](first-operation-proof.md). For the operator daily rhythm, see [`docs/operator-loop.md`](operator-loop.md). For the trial-specific runbook, evidence format, and redaction policy, see [`docs/live-trial-runbook.md`](live-trial-runbook.md).

> **Command convention:** All CLI commands in this document use `-c config/config.json`, matching the `package.json` scripts generated by `narada init-repo`. If your ops repo places config elsewhere, adjust the path accordingly.

---

## Proof Scope

**Operation:** Support mailbox (`help@global-maxima.com`)  
**Charter:** `support_steward`  
**Posture:** `draft-only` with `require_human_approval: true`  
**Proof path:** Inbound message Рєњ fact Рєњ work item Рєњ evaluation Рєњ foreman decision Рєњ outbound command Рєњ draft creation Рєњ operator approval Рєњ send submission Рєњ reconciliation confirmation

---

## Proof Stages

### Stage 0: Prerequisites

**Before starting the proof, verify:**

```bash
# 1. Daemon is running
narada doctor -c config/config.json

# 2. Graph API is reachable
narada preflight help@global-maxima.com -c config/config.json

# 3. Charter runtime is healthy
narada ops -c config/config.json
```

**Expected state:** `overall: healthy`, `preflight: pass`, `charterRuntimeHealthy: true`.

---

### Stage 1: Inbound Thread Selected

**Trigger:** A new email arrives in the monitored mailbox. The daemon sync loop picks it up.

**What must exist:**

| Evidence | Location |
|----------|----------|
| Sync cursor advanced | `<rootDir>/state/cursor.json` |
| Message record persisted | `<rootDir>/messages/<message-id>/record.json` |
| Fact stored | `facts` table in coordinator DB |
| Context record created/updated | `context_records` table |
| Work item opened | `work_items` table with `status = 'opened'` |

**Canonical inspection:**

```bash
# Check sync happened
narada status -c config/config.json

# Check work items opened
narada ops -c config/config.json

# Or query directly
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT work_item_id, context_id, status, opened_at FROM work_items WHERE status = 'opened';"
```

**Pass criterion:** A `work_item` row exists with `status = 'opened'` for the new message's `context_id`.

---

### Stage 2: Evaluation Produced

**Trigger:** Scheduler leases the work item; charter evaluates the context.

**What must exist:**

| Evidence | Location |
|----------|----------|
| Work item `leased` Рєњ `executing` Рєњ `resolved` | `work_items` table |
| Execution attempt completed | `execution_attempts` table |
| Evaluation persisted | `evaluations` table |

**Canonical inspection:**

```bash
# Show the latest evaluation
narada show evaluation <evaluation-id> --operation help@global-maxima.com -c config/config.json

# Or query directly
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT evaluation_id, charter_id, outcome, summary, analyzed_at \
   FROM evaluations ORDER BY analyzed_at DESC LIMIT 1;"
```

**Pass criterion:** An `evaluation` row exists with:
- `outcome = 'action_proposed'` or `'no_action'`
- `proposed_actions` contains a `draft_reply` or `send_reply` action (if action proposed)
- `summary` is non-empty

---

### Stage 3: Outbound Draft Created

**Trigger:** Foreman resolves the work item; `DefaultForemanFacade` creates a foreman decision; `OutboundHandoff` creates an outbound command. For `draft-only` posture with `require_human_approval: true`, the decision status is `pending_approval` and the outbound command is created with `status = 'pending'`.

**What must exist:**

| Evidence | Location |
|----------|----------|
| Foreman decision | `foreman_decisions` table |
| Outbound command | `outbound_handoffs` table |
| Outbound version | `outbound_versions` table |
| Outbound transition logged | `outbound_transitions` table |

**Canonical inspection:**

```bash
# Show the decision
narada show decision <decision-id> --operation help@global-maxima.com -c config/config.json

# Check outbound commands pending
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, action_type, status, created_at \
   FROM outbound_handoffs WHERE status = 'pending' ORDER BY created_at DESC;"

# Check the version payload
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT v.outbound_id, v.version, v.subject, v.body_text \
   FROM outbound_versions v \
   JOIN outbound_handoffs h ON v.outbound_id = h.outbound_id \
   WHERE h.status = 'pending' ORDER BY v.created_at DESC LIMIT 1;"
```

**Pass criterion:**
- `foreman_decisions` row exists with `approved_action = 'draft_reply'` or `'send_reply'`
- `outbound_handoffs` row exists with matching `outbound_id`, `status = 'pending'`
- `outbound_versions` row exists with the draft payload (subject, body_text, to)

---

### Stage 4: Draft Approved for Send

**Trigger:** Operator reviews the draft and approves it.

**For `draft_reply` actions:** Approval is not required. The `SendReplyWorker` will create the Graph draft directly when it processes the `pending` command.

**For `send_reply` actions with `require_human_approval: true`:** Operator must explicitly approve.

```bash
# Review the draft payload first
narada show decision <decision-id> --operation help@global-maxima.com -c config/config.json

# Approve for send
narada approve-draft-for-send <outbound-id> -c config/config.json
```

**What must exist after approval:**

| Evidence | Location |
|----------|----------|
| Command status `approved_for_send` | `outbound_handoffs.status` |
| `approved_at` timestamp set | `outbound_handoffs.approved_at` |
| Transition logged | `outbound_transitions` |
| Operator action audit record | `operator_action_requests` table |

**Canonical inspection:**

```bash
# Verify approval
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, status, approved_at FROM outbound_handoffs WHERE outbound_id = '<outbound-id>';"

# Verify audit trail
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT action_type, target_id, requested_at, status \
   FROM operator_action_requests WHERE target_id = '<outbound-id>';"
```

**Pass criterion:** `status = 'approved_for_send'` and `approved_at` is non-null.

---

### Stage 5: Send Submitted

**Trigger:** The dedicated `SendExecutionWorker` picks up the approved command, verifies the managed draft integrity, and performs the Graph send.

**State progression:**
1. `SendExecutionWorker` transitions command to `sending`
2. Graph API send succeeds Рєњ `submitted`
3. `submitted_at` is set

(For `draft_reply` actions, `SendReplyWorker` creates the Graph draft and reaches `draft_ready` directly; no separate execution worker is needed.)

**What must exist:**

| Evidence | Location |
|----------|----------|
| Command status `submitted` | `outbound_handoffs.status` |
| Managed draft record | `managed_drafts` table |
| Graph draft ID | `managed_drafts.draft_id` |
| `submitted_at` timestamp | `outbound_handoffs.submitted_at` |
| Transition logged | `outbound_transitions` |

**Canonical inspection:**

```bash
# Check submitted status
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, status, submitted_at FROM outbound_handoffs WHERE status = 'submitted';"

# Check managed draft
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, draft_id, internet_message_id, created_at FROM managed_drafts;"

# Check transitions
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, from_status, to_status, transition_at \
   FROM outbound_transitions WHERE outbound_id = '<outbound-id>' ORDER BY transition_at;"
```

**Pass criterion:**
- `status = 'submitted'`
- `managed_drafts` row exists with `draft_id` (Graph draft ID)
- Transition chain shows `approved_for_send Рєњ sending Рєњ submitted` for approved sends, or `pending Рєњ draft_creating Рєњ draft_ready Рєњ confirmed` for draft-only actions

---

### Stage 6: Reconciliation Confirms Sent Result

**Trigger:** Next sync cycle observes the sent message in the mailbox (inbound reconciliation).

**What must exist:**

| Evidence | Location |
|----------|----------|
| Command status `confirmed` | `outbound_handoffs.status` |
| `confirmed_at` timestamp | `outbound_handoffs.confirmed_at` |
| New inbound message with `X-Outbound-Id` header | `<rootDir>/messages/<new-message-id>/record.json` |
| Transition logged | `outbound_transitions` |

**Canonical inspection:**

```bash
# Check confirmed status
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, status, confirmed_at FROM outbound_handoffs WHERE status = 'confirmed';"

# Look for the sent message in the message store
ls ./mailboxes/help-global-maxima/messages/

# Verify the message header links back to the outbound command
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT outbound_id, draft_id, internet_message_id FROM managed_drafts WHERE outbound_id = '<outbound-id>';"
```

**Pass criterion:**
- `status = 'confirmed'`
- `confirmed_at` is non-null
- A new message appears in the message store (the sent reply, now visible in the mailbox via sync)
- The `internet_message_id` in `managed_drafts` matches the sent message

                                         Рћѓ                      Рћѓ
                                         РћћРћђРєњ confirmed          sending Рєњ submitted Рєњ confirmed

                                         РћћРћђРєњ cancelled          Рћѓ

```
pending Рєњ draft_creating Рєњ draft_ready Рєњ approved_for_send Рєњ sending Рєњ submitted Рєњ confirmed
```

For the **live proof path** with `draft_reply` (no approval needed):

```
pending Рєњ draft_creating Рєњ draft_ready Рєњ confirmed
```

| [`docs/runbook.md`](runbook.md) | Troubleshooting and setup |
| Integration tests | Fixture-backed pipeline proofs |

### Private Ops Repo

| Artifact | What It Contains |
|----------|------------------|
| `config/config.json` | Live mailbox IDs, scope configuration |
| `.env` | Graph API credentials, charter API keys |
| `mailboxes/<scope>/` | Real message data, cursors, health files |
| `logs/` | Daemon logs with real message IDs |
| Live proof execution log | Transcript of running this proof against real data |

**Rule:** No private message bodies, Graph message IDs, email addresses of real correspondents, or credentials may be committed to the public repo. The public repo contains the *contract* and *method*. The private ops repo contains the *evidence*.

## Operator Runbook: Execute the Live Proof

This is the minimal runbook for a real operator to exercise the full live proof.

### Pre-Flight

```bash
# 1. Verify daemon health
narada doctor -c config/config.json

# 2. Verify Graph connectivity
narada preflight help@global-maxima.com -c config/config.json

# 3. Verify no stuck items from previous runs
narada ops -c config/config.json
```

### Step 1: Trigger Inbound (Send a Test Email)

Send an email to `help@global-maxima.com` from an external account. The subject and body should be clearly identifiable as the test case (e.g., "Live proof test Рђћ please ignore").

### Step 2: Sync

```bash
# Force a sync cycle (or wait for daemon to sync automatically)
narada sync --mailbox help@global-maxima.com -c config/config.json
```

### Step 3: Verify Inbound Processing

```bash
# Check that a work item was opened
narada ops -c config/config.json

# Verify the message was stored
ls ./mailboxes/help-global-maxima/messages/
```

### Step 4: Wait for Evaluation

The daemon scheduler will lease the work item and run the charter. This typically takes 5РђЊ30 seconds after sync completes.

```bash
# Watch for the evaluation
narada ops -c config/config.json

# Or poll for evaluations
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT evaluation_id, outcome, summary FROM evaluations ORDER BY analyzed_at DESC LIMIT 1;"
```

### Step 5: Inspect the Proposed Draft

```bash
# Find the latest decision
narada ops -c config/config.json

# Deep-dive into the decision
narada show decision <decision-id> --operation help@global-maxima.com -c config/config.json

# Inspect the outbound version payload
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT subject, body_text FROM outbound_versions ORDER BY created_at DESC LIMIT 1;"
```

### Step 6: Approve for Send

```bash
narada approve-draft-for-send <outbound-id> -c config/config.json
```

If the draft is unsatisfactory, reject it instead:

```bash
narada reject-draft <outbound-id> --rationale "Live proof: draft rejected" -c config/config.json
```

### Step 7: Verify Send Submission

Wait for the outbound worker to process the approved command (typically within the next daemon tick, 5РђЊ30 seconds).

```bash
# Check status
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT status, submitted_at FROM outbound_handoffs WHERE outbound_id = '<outbound-id>';"

# Check managed draft
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT draft_id, internet_message_id FROM managed_drafts WHERE outbound_id = '<outbound-id>';"
```

### Step 8: Verify Reconciliation

Run another sync cycle to trigger inbound reconciliation:

```bash
narada sync --mailbox help@global-maxima.com -c config/config.json
```

Then verify confirmation:

```bash
# Check confirmed status
sqlite3 ./mailboxes/help-global-maxima/.narada/coordinator.db \
  "SELECT status, confirmed_at FROM outbound_handoffs WHERE outbound_id = '<outbound-id>';"

# Check that a new message appeared (the sent reply)
ls ./mailboxes/help-global-maxima/messages/ | wc -l
```

### Step 9: Log the Proof

Record in your private ops log:

```
Live Graph Proof Рђћ <date>
- Test email subject: "..."
- Work item ID: ...
- Evaluation ID: ...
- Decision ID: ...
- Outbound ID: ...
- Graph draft ID: ...
- Sent message ID: ...
- Status: confirmed
- Operator: <name>
```

## Non-Goals

- This proof does not exercise autonomous send (`require_human_approval: false`). The default posture is `draft-only` with approval required.
- This proof does not test multi-vertical operations (timer, webhook).
- This proof does not test attachment handling, HTML body rendering, or CC/BCC scenarios.
- This proof does not test failure recovery (retry_wait, failed_terminal). Those are covered by fixture tests and the runbook.
- This proof does not store private evidence in the public repo.
