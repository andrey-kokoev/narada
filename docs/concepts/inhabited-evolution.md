# Inhabited Evolution

Inhabited Evolution is Narada's construction mode for building doctrine, tooling, and substrate from real operations.

It is not ordinary delegation, chat, brainstorming, or speculative architecture. It is a governed co-evolution loop in which a human Operator, an AI architect, and a durable trace substrate improve the system while the originating operation remains live enough to test the lifted form.

## Core Path

```text
situated ops
  -> friction
  -> local construction
  -> trace/evidence
  -> invariant extraction
  -> re-instantiation
  -> doctrine/tooling/substrate lift
```

## Roles

| Role | Function | Authority Limit |
| --- | --- | --- |
| Operator | Names pressure, corrects direction, grants authority, rejects unearned structure. | Operator correction is authority-bearing input, not conversational decoration. |
| AI architect | Forms hypotheses, proposes structure, edits bounded artifacts, detects incoherence. | Intelligence proposes and constructs; it does not silently grant itself authority. |
| Trace substrate | Preserves tasks, evidence, diffs, commands, failures, residuals, and decisions. | Trace explains and constrains future construction; it is not a substitute for crossing authority. |

## Rules

1. Build what the operation has earned.
2. Lift only when the originating case can still run through the lifted form.
3. Record unearned structure as possibility, proposal, or residual.
4. Treat Operator correction as part of the control loop.
5. Distinguish inhabited synthesis from ordinary delegation or chat.
6. Preserve Intelligence-Authority Separation.
7. Push back against unearned symmetry and speculative machinery.

## Narada Reading

In Aim / Site / Cycle / Act / Trace terms:

| Inhabited Evolution Element | Narada Reading |
| --- | --- |
| Situated operation | Aim-at-Site binding under live pressure |
| Friction | Evidence-bearing signal from a Cycle, Operator correction, failed command, rough surface, or residual |
| Local construction | Bounded task/chapter change made in the current Site |
| Trace/evidence | Task evidence, verification runs, command results, reviews, observations, commits |
| Invariant extraction | Doctrine-level rule that remains true outside the originating case |
| Re-instantiation | The originating case is rerun through the lifted form |
| Doctrine/tooling/substrate lift | Durable change to semantics, CLI, package boundaries, Site substrate, or construction law |

## Admission Test For Lifting

A proposed lift is earned only when all of these are true:

- The originating friction is concrete and traceable.
- The proposed form eliminates or narrows that friction.
- The originating case can still run through the new form.
- Residual ambiguity is recorded instead of hidden.
- Authority crossings remain explicit.

If any condition fails, the material stays as a proposal, possibility, residual, or local workaround. It does not become doctrine.

## Anti-Collapse Rules

| Collapse | Why It Is Wrong | Correct Form |
| --- | --- | --- |
| Chat becomes authority | Conversation alone is not a durable crossing artifact. | Operator instruction enters through governed command, task, inbox item, or recorded decision. |
| AI synthesis becomes decision | Intelligence can see patterns but cannot grant itself authority. | Synthesis becomes proposal, patch, evidence, or doctrine only after admission. |
| One case becomes universal law | Local pressure can overfit. | Extract invariant only after the originating case survives the lifted form. |
| Symmetry replaces need | Beautiful parallel machinery can be unearned. | Build the smaller crossing or operator surface that current friction justifies. |
| Trace becomes decoration | Unused evidence cannot constrain future work. | Trace must be inspectable and capable of changing future Cycles. |

## Relationship To Zones

Inhabited Evolution is not itself a zone. It is a construction posture spanning zones.

It uses zones and crossings to prevent shortcuts:

- Operator pressure enters through Operator Input or task/inbox authority surfaces.
- Command and test execution pass through CEIZ/TIZ-style governed execution paths.
- Evidence is admitted before lifecycle closure.
- Observations are bounded before they are shown to the Operator.
- Doctrine changes are committed as durable artifacts.

## Practical Agent Rule

When building Narada itself, an agent should ask:

```text
What friction did this operation earn, what invariant does it expose, and can the originating case run through the lifted form?
```

If the answer is unclear, do not invent architecture. Record the uncertainty as a residual or proposal and keep the next change local.

Agents are expected to push back when a proposed move is not earned by the operation or moves against Narada's spirit. This is not refusal for its own sake; it is protection of the construction loop. The correct pushback shape is:

- name the concrete friction or invariant that is missing;
- identify whether the proposal should remain possibility, proposal, residual, or rejected path;
- prefer the smallest local construction that lets the originating case keep running;
- avoid adding machinery merely to complete a symmetry;
- route any authority-bearing change through a governed artifact.
