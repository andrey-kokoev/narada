# Governed Locus Federation

Governed Locus Federation is the Narada doctrine for continuity across multiple evolving loci without collapsing them into one authority domain.

It names the parent pattern behind:

- Plural Embodiment, Singular Authority;
- Site lifecycle transformations;
- Site provenance lineage;
- Site pub/sub influence;
- governed admission through inbox/task/knowledge/tool boundaries;
- re-instantiable Site memory.

## Definition

A governed locus federation is a set of Narada-capable loci that can exchange signals, templates, knowledge, tools, evidence, and transformation lineage while each governed mutation remains owned by exactly one declared authority locus.

The compact rule is:

```text
Federate awareness.
Govern admission.
Preserve singular mutation authority.
Record lineage.
Prove re-instantiation before lifting.
```

## Boundary Distinctions

| Concern | Meaning | Must Not Collapse Into |
| --- | --- | --- |
| Embodiment topology | Many shells, clones, agents, daemons, UIs, and projections can present Narada. | Mutation authority. |
| Authority routing | Each mutation class resolves to one declared locus. | Convenience of current caller. |
| Site lifecycle | Sites can clone, fork, split, absorb, migrate, re-instantiate, or archive. | Raw folder copy or registry editing. |
| Provenance lineage | Append-only events record where loci and authority effects came from. | Decorative diagrams or unverifiable history. |
| Pub/sub influence | Sites can publish and subscribe to typed signals. | Automatic admission, replication, or ownership. |
| Governed admission | Incoming material remains inert until promoted by the receiving locus. | The sender's intent. |
| Verifiable envelope trust | Messages may carry authenticity, integrity, confidentiality, and provenance evidence. | Mutation authority or admission. |
| Re-instantiable memory | Templates and lifted machinery must rerun the originating case. | Abstract symmetry or unearned platform structure. |

## Admission Rule

A federated signal from one locus to another is never self-executing.

It may enter as:

- inbox envelope;
- task candidate;
- knowledge candidate;
- tool candidate;
- Site config change proposal;
- template candidate;
- authority migration proposal;
- lineage event candidate.

The receiving locus owns admission. The sending locus may provide evidence and recommendation, but it does not mutate receiver truth.

## Anti-Collapse Rules

| Collapse | Correct Form |
| --- | --- |
| Federation becomes automatic replication | Use typed signals plus governed admission. |
| Subscription becomes ownership | Mark subscription/publication edges as influence-only. |
| Clone becomes second authority | Declare read-only, forwarding, or authority-migration mode. |
| Template becomes proof | Require re-instantiation evidence. |
| Lineage graph becomes authority | Treat graph as projection over append-only lineage events. |
| Parent concept becomes speculative platform | Keep unimplemented parts as proposals until operations earn them. |

## Operational Reading

Governed Locus Federation is not a new mandatory runtime layer. It is an organizing doctrine for Narada as it becomes a network of Sites, embodiments, and reusable machinery.

Implementation should remain incremental:

1. preserve target-locus safety;
2. use canonical inbox for typed intake;
3. add read-only vocabularies before mutation commands;
4. require transformation plans before Site mutation;
5. write lineage events before changing authority;
6. admit pub/sub as influence-only until receiving loci promote signals;
7. lift templates only after re-instantiation proof.
