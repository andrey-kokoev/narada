## Runtime and ownership model

The intended factorization is:

User launcher -> Narada CLI planning/execution -> NARS agent-runtime-server -> operator surface

The old PowerShell launcher remains a user-facing entry point, but the launch plan and execution logic have been moved into Narada CLI. A launcher failure that says the CLI dist is stale usually means the source changed without rebuilding the CLI package. The usual remedy is a focused build from D:/code/narada, not an ad hoc edit to the PowerShell shim.

NARS owns:

- session identity and lifecycle;
- event ordering and durable session evidence;
- health and attachment status;
- provider turn execution;
- authority zones and command admission;
- MCP fabric construction and runtime capabilities;
- projection/event protocol methods.

Agent Web UI owns:

- browser transport and attachment;
- browser-side session store and replay projection;
- Vue operator-surface rendering;
- view selection and readable event projection;
- browser-local input behavior and browser preferences;
- browser-facing panels that project NARS-advertised surface data.

Agent Web UI does not own runtime construction, provider execution, MCP hosting, session authority, or slash-command execution. This is stated in its package metadata and README. Provider/model/thinking changes displayed in the UI are NARS session actions; they are not browser preferences.

agent-cli and agent-web-ui should be able to attach to the same healthy NARS session in parallel. The session identifier still commonly has a historical `carrier_` prefix. That name is a compatibility-era artifact in the surrounding launcher/runtime vocabulary; do not assume it means the operator surface owns the runtime. There has been an ongoing effort to remove or replace the Carrier term, so inspect the current contract before introducing new naming.

## Current UI shape

The UI has a fixed shell/header and composer around a scrollable event area. Current concepts include:

- Chat/conversation, operations, diagnostics, and raw projection regimes.
- Event-based rendering, with NARS lifecycle events as the source of truth.
- Agent activity indicators such as queued/thinking/responding, derived from runtime events.
- Operator input projection through the NARS session protocol.
- Markdown, code/render tabs, tables, lists, fenced code, and Mermaid rendering in appropriate event content.
- MCP, SOP, Site, and other authority/panel projections where NARS advertises the relevant affordance.
- A right-side full-height panel pattern for MCP/SOP/site details.
- Collapsible status/box rows and collapsible operator queue state.
- Browser-local preferences under fully qualified `narada:agent-web-ui:<key>` keys, owned by the feature that uses them.
- Session replay/attach through a local or Cloudflare projection endpoint.
- Copyable operator-facing tokens and diagnostics where the UI makes the token identity explicit.

Important semantic boundaries:

- Conversation view should show human-readable projected messages, not raw protocol event objects.
- Operations/diagnostics/raw views may show more lifecycle and protocol detail.
- Do not pretend that every NARS event is a token stream. NARS is event-based. Activity indicators may show that a turn is active, but the UI should not fabricate partial assistant text unless the protocol explicitly supplies a meaningful partial event.
- Health polling/transport state belongs in status UI and should not spam the conversation transcript.
- A submitted operator message can have both an input-submitted event and an admitted/user-message event. These are not automatically two visible conversation messages. Projection deduplication must use event identity/render disposition, not crude text concatenation.
- Assistant lifecycle/tool events can contain separate assistant messages. They should remain visibly separated when the projection says they are separate; concatenating them into one giant string makes the conversation look incoherent.
- Transient activity rows must be retired or updated when the turn reaches a terminal state. A stale "Waiting for agent..." or "queued" row is a state-model bug, not a CSS problem.
- Scroll-to-bottom should happen only when the operator is at/near the bottom or has just submitted input. Arriving background events must not forcibly yank an operator who is reading history.

## Historical failure modes worth checking before claiming completion

1. Dist/source drift:
   - Launcher and UI can execute built artifacts rather than source.
   - Rebuild the exact package after source edits.
   - If the launcher reports `narada_cli_dist_stale`, rebuild the CLI from the Narada root.
   - If the UI looks unchanged, verify the launched server's artifact path and rebuild before changing code again.

2. Runtime/session confusion:
   - An agent-web-ui attach without an explicit session can find multiple sessions for one agent.
   - A stale or unhealthy discovered session must produce a readable refusal and next action.
   - A new Web UI should not silently create a second unrelated NARS session when the operator intended an attachment.
   - A single NARS session can have multiple operator surfaces; surface processes are not runtime authority.

3. Protocol drift:
   - The unsupported `session.surface.affordances` method has regressed multiple times in the past.
   - Any test intended to prevent this must exercise a real NARS protocol path or the real runtime adapter, not only a mocked method table or a local shim.
   - Check capability negotiation before calling optional methods.
   - When a surface is unavailable, render "not advertised" or a useful unavailable state rather than `[object Object]` or an unexplained raw protocol error.

4. Projection duplication:
   - Do not render the same logical input or assistant message twice because two evidence events share text.
   - Treat stream fragments, completed assistant messages, user-message admission, and operator input submission according to their explicit projection disposition.
   - Keep a visible separator/pause cue between genuinely separate assistant message events.

5. Activity-state drift:
   - "Waiting for agent...", "queued", "thinking", and "responding" are transient state projections.
   - They must be reconciled with turn start/completion/failure and must not remain after completion.
   - Health heartbeat samples belong outside Conversation and should be summarized or hidden from the transcript.

6. View reactivity:
   - The selected view is a reactive projection choice. On initial load, replayed events must be classified according to the selected view before rendering.
   - Changing Chat/Operations/Diagnostics/Raw must recompute the visible projection, not just alter a label.
   - Chat should remain the default view unless the current product decision changes.

7. Authority boundaries:
   - UI buttons/selects may request an admitted NARS affordance.
   - They must not call MCP servers directly or mutate durable state outside the authority zone.
   - Site/MCP/SOP panels are observation/projection surfaces unless a deliberate protocol affordance exists.
   - Browser localStorage is for presentation preferences, never session authority or durable event state.
