## Current Identity

NARS is Narada's authority runtime for one governed session. It owns session control, durable session evidence, turn admission, queue state, recovery, and lifecycle. Attached terminal, TUI, web, and remote surfaces are projections unless an explicit authority transfer says otherwise.

Ownership boundaries:

- `@narada2/nars-session-core`: session state, durable event journal, input queue/admission, turn lifecycle, recovery, artifacts, event attachment, session index.
- `@narada2/agent-runtime-server`: stable process entrypoint, session binding, JSONL/HTTP/WebSocket projections, health, event subscription, lifecycle hooks, runtime-host state, artifact HTTP projection.
- `@narada2/nars-provider-runtime`: provider request shaping, dispatch, owned subprocesses, provider invocation evidence.
- `@narada2/nars-capability-gateway`: MCP hosting, capability admission, tool execution lifecycle/evidence.
- Client packages: input/rendering/attach UX only. They must not recreate runtime, provider, MCP, or session authority.

## Narrow Control Contract

The local session-core control contract is:

- `session.submit`
- `session.health`
- `session.recovery`
- `session.cancel`
- `session.close`

`session.events.subscribe` and `session.events.read` are event-stream transport controls, not extra turn controls.

`runtime.intelligence.reconfigure` is owned by the runtime server and remains separate from session-core control because it changes provider execution policy, not session authority or turn history. It must be refused while a turn is active or operator input is pending; credentials never enter the request or evidence.

Historical `conversation.*`, broad observer/status/operations, authority-transition, affordance, and panel-summary methods are not local session-core methods. Named adapters may retain that vocabulary only outside the local boundary and must translate into the narrow contract.

## Durable State and FSMs

- Turn: `accepted -> contextualized -> evaluating -> tool_requested -> tool_admitted -> executing -> reconciling -> evaluating -> completed|blocked|interrupted|failed|refused`.
- Input admission: `accepted -> queued -> admitted`; explicit `held` returns to `queued`; shutdown moves pending inputs to `abandoned`.
- Recovery attempt: `requested -> claimed -> replaying -> reconciled -> completed`; retries create a new attempt.
- Event attachment: `requested -> replaying -> live -> closing -> closed`; sender failure is terminal and removes the subscriber.
- Provider invocation: `requested -> validated -> shaped -> dispatched -> receiving -> completed`, with explicit refused/failed/interrupted terminals.
- Runtime host: bound/serving lifecycle is separate from session, turn, provider, tool, and input-admission state.
- Artifact content is readable only while its session-core lifecycle record is active.

State transitions are durable evidence. Do not add implicit replay, hidden retries, or client-owned lifecycle state.

## Identity and Storage Rules

- Canonical session environment field: `NARADA_NARS_SESSION_ID`.
- Compatibility fallback: `NARADA_RUNTIME_SESSION_ID`.
- Legacy-only alias: `NARADA_CARRIER_SESSION_ID`; do not present it as the primary session identity.
- Session storage is resolved with `@narada2/site-paths`, under `<siteAuthorityRoot>/crew/nars-sessions/<session-id>/`.
- `events.jsonl`, control-sideband evidence, heartbeat, health, and index projections are evidence/read models around the session authority. Do not create a parallel queue or mutate session state directly from a client.
