# Narada Missing Capabilities Frontier

> Canonical ranked statement of what Narada is still materially missing after the crossing-regime and taxonomy tranches.
>
> This is not a generic wishlist. It is the current frontier of missing proof, missing closure, and missing self-governance needed for Narada to become a fluid governed operation.

**Date:** 2026-04-23

**Inputs**
- Crossing-regime first-class chapter (`495–500`)
- Zone-template / regime-kind taxonomy chapter (`503–506`)
- Task-governance hardening (`463`, `474`, `486`, `501`, `507`, `508`, `509`)
- Email-marketing operation and live dry-run chapter (`387–405`)
- macOS/Linux Site materialization chapters (`431–442`)
- Narada self-build / construction-loop work (`410–415`, `464`, `470+`)

---

## 1. Current Judgment

Narada is no longer primarily missing doctrine. It is primarily missing:

1. **one real supervised live proof in routine use,**
2. **closure of the remaining old substrate/article drift,**
3. **extraction of self-governance so the operator is no longer the hidden scheduler,**
4. **a second traveling operation/vertical proof,**
5. **and a stronger agent/runtime model for Narada governing itself.**

What Narada is **not** mainly missing now:

- more vocabulary work,
- more semantic crystallization for its own sake,
- broader generic abstraction,
- or stricter closure policing for already-solved governance patterns.

Those have become secondary. The current bottleneck is **operational proof and self-governing execution**.

---

## 2. Ranked Missing Capabilities

### Rank 1 — Supervised Live Proof As Routine Path

**Why it matters**

Narada has strong fixture-backed structure, but its strongest product-shaped line still lacks a normal, practiced live proof path. The clearest surface is the email-marketing live dry run (`403–405`).

**What is missing**

- a repeatable supervised live Cycle over real input,
- inspectable governed output,
- operator proof of no forbidden side effect,
- and closure of that live path as something Narada can actually do, not just simulate.

**Why this is first**

Without one real supervised proof in ordinary use, Narada remains over-validated in doctrine and under-validated in reality.

---

### Rank 2 — Remaining Old Chapter / Artifact Drift

**Why it matters**

Some old chapter lines still contain unresolved root tasks or closure drift. That means the governed artifact layer still has islands where reality and task state do not fully align.

**Current concrete examples**

- `403–405` remains open and operator-gated.
- macOS root contract task `431` remains genuinely open.
- Linux root contract task `437` remains genuinely open.

**What is missing**

- either actual completion of the unresolved root contract tasks,
- or an explicit supersession/repair decision if the later implementation truly replaced them.

**Why this is second**

Narada’s task-governed self-model cannot be considered clean while old substrate chapters still contain unresolved root artifacts.

---

### Rank 3 — Self-Governance Extraction

**Why it matters**

Narada can now govern tasks, assignments, reviews, closure, evidence, recommendations, and chapter state. But the real development loop still relies too much on the human operator to decide:

- who works next,
- which tranche matters now,
- how to promote recommendation into assignment,
- and how to steer the buildout when multiple partially-complete lines exist.

**What is missing**

- a more complete governed self-build loop,
- bounded recommendation → assignment promotion,
- better principal/agent runtime integration,
- and less hidden reliance on the human as scheduler-of-last-resort.

**Why this is third**

The operator toll is rising. This is now a real constraint on Narada’s own becoming.

---

### Rank 4 — Agent / Runtime First-Class Modeling

**Why it matters**

Narada still does not fully model the architect-operator + agent swarm pattern that is actually being used to build Narada.

**What is missing**

- clearer first-class agent/runtime state modeling,
- a better bridge between task governance and principal runtime,
- and an extraction of the real pair/roster architecture into Narada itself.

**Why this is fourth**

Until Narada can structurally model the pattern currently used to build it, its self-application remains partial.

---

### Rank 5 — A Second Traveling Operational Vertical

**Why it matters**

Mailbox/email-marketing is one strong real vertical, but Narada still wants another operation family that proves the topology travels.

**What is missing**

- one more real operation type or vertical,
- with different source/effect characteristics,
- still governed through the same zone/crossing structure.

**Why this is fifth**

This is how Narada proves it is a portable control grammar rather than a refined mailbox system.

---

### Rank 6 — Unattended / Semi-Attended Multi-Site Operational Proof

**Why it matters**

Narada is coherent locally and increasingly coherent by substrate, but not yet proven as a fluid unattended or semi-attended multi-site operational system.

**What is missing**

- stronger unattended proof,
- more real Site/operator loop practice,
- and better proof that Narada remains governed without constant babysitting.

**Why this is sixth**

This matters, but it should be built on ranks 1–4 rather than pursued as pure infrastructure.

---

## 3. Negative Guidance

The following should **not** be treated as primary next missing capabilities:

- another doctrinal crystallization pass without new pressure,
- generic runtime abstraction from the new taxonomies,
- provenance-safe-by-construction across every path before a consumer exists,
- or more closure tightening on already-governed task surfaces unless a new bypass is found.

These may still happen, but they are not the frontier.

---

## 4. What This Means Operationally

If Narada asks “what should we do next?”, the disciplined answer is:

1. land one real supervised live path,
2. clean the remaining old chapter/root-task drift,
3. push self-governance extraction,
4. make the agent/runtime architecture more first-class,
5. then prove a second traveling operation.

That is the current missing-capabilities frontier.
