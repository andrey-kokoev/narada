# Becoming Better Faster — Chapter Priority

> Canonical prioritization of the current chapter backlog for making Narada better faster, not just broader.

**Date:** 2026-04-24

**Primary inputs**
- `.ai/decisions/20260423-next-capability-frontier-chapters.md`
- `.ai/decisions/20260423-narada-missing-capabilities-frontier.md`

---

## Judgment

Narada is no longer primarily coherence-poor. It is now more often **execution-lagged**: doctrine, boundaries, and anti-smear distinctions are increasingly clear, but too much of that clarity still waits to become routine executable surface.

So the right chapter priority is not "what adds the most new scope?" It is:

> **what reduces hidden authority, operator drag, and execution friction fastest without reintroducing smear or duplication.**

---

## Ranked Priority

### 1. Task Lifecycle State Authority Migration (`546–550`)

**Why first**

This is the highest-leverage coherence and execution chapter currently on disk.

Narada still repeatedly suffers the same failure mode:
- agents complete substantive work,
- task markdown is mutated directly,
- governed lifecycle provenance is bypassed,
- and the system then has to repair artifact truth after the fact.

This chapter addresses the root cause rather than more symptoms.

**Expected gain**
- less lifecycle ambiguity,
- less operator repair work,
- stronger auditability,
- lower risk of dual-truth between state and artifact.

**Better faster contribution**

Very high. This improves Narada's own governance substrate and reduces recurring friction immediately.

---

### 2. Operator Console Substrate Completion (`536–540`)

**Why second**

Narada already has an Operator Console layer, but it is uneven by substrate:
- Windows is strongest,
- Cloudflare lacks deep observation parity,
- Linux remains only partially meaningful.

Completing this chapter makes the current operator surface materially more real without requiring a new conceptual framework.

**Expected gain**
- stronger real operability on Cloudflare/Linux,
- less substrate asymmetry,
- better routine inspection/control,
- more trustworthy operator posture across Sites.

**Better faster contribution**

High. This improves real use now.

---

### 3. Workbench v0 Build (`526–529`)

**Why third**

This chapter reduces manual relay work in Narada self-build by giving the operator a browser workbench over governed state. It is already an implementation line rather than a shaping line, which is good.

It ranks below the two above because:
- task authority migration fixes a deeper recurring substrate flaw,
- console completion improves already-existing product operability,
- and workbench v0 is primarily for Narada-building-Narada, not general operation first.

**Expected gain**
- less chat/terminal choreography,
- better swarm visibility,
- faster self-build supervision.

**Better faster contribution**

High for development runtime, moderate for Narada product runtime.

---

### 4. Mail Connectivity Generalization (`531–535`)

**Why fourth**

This chapter is strategically important because it lets the mail vertical travel beyond Microsoft Graph. But it is still a boundary/generalization chapter, not immediate execution hardening.

It expands Narada's reachable world, but it does not reduce today's highest operator/coherence friction as directly as the first three.

**Expected gain**
- cleaner provider-agnostic mail boundary,
- Gmail / generic provider readiness,
- less Graph-specific lock-in.

**Better faster contribution**

Medium-high strategically, lower immediately.

---

### 5. Messaging Connectivity Family Boundary (`541–545`)

**Why fifth**

This is a healthy expansion line and an important anti-smear move, but it is still farther from immediate operational payoff than the others.

It helps Narada become broader and more portable, but not necessarily better faster in the near term.

**Expected gain**
- clear separation between messaging and mail families,
- Telegram-ready conceptual posture,
- cleaner future provider growth.

**Better faster contribution**

Medium. Important, but not the first execution bottleneck.

## Operational Reading

If Narada asks:

> "What should we do to become better faster?"

the current disciplined answer is:

1. fix task lifecycle authority,
2. complete Operator Console substrate parity,
3. finish the local workbench,
4. then broaden connectivity families.

That is the best current order for improving Narada's actual capability density without sacrificing coherence.
