# Task 255: Init and Setup Path Hardening

## Chapter

Product Surface Coherence

## Context

A new user has two conflicting init paths (`init` vs `init-repo`), incomplete credential checking, and gaps in the first-run experience. The `want-mailbox` CLI does not expose options the library already supports. The daemon does not detect the ops-repo config location.

## Goal

Unify and harden the init/setup path so a new user can go from zero to a running operation with minimal friction and no manual config editing.

## Required Work

### 1. Unify Init Commands

In `packages/layers/cli/src/commands/config.ts` and `config-interactive.ts`:
- Deprecate `narada init` (non-interactive legacy config generator). Print a redirect message: "Use `narada init-repo <path>` instead."
- Make `narada init --interactive` ops-repo-aware: write to `./config/config.json` inside the current directory (or create `./config/` if absent), using the modern `scopes[]` shape.
- Remove the legacy `mailbox_id`-at-root default from `config.ts`.

### 2. Extend `want-mailbox` CLI

In `packages/layers/cli/src/main.ts`:
- Add `--graph-user-id <id>` to `want-mailbox`.
- Add `--folders <list>` (comma-separated) to `want-mailbox`.
- Add `--data-root-dir <path>` to `want-mailbox`.
- Update help text to describe each option.

### 3. Harden Preflight

In `packages/ops-kit/src/readiness/collect.ts`:
- Before reporting "fail" on Graph credentials, check:
  1. Env vars (`GRAPH_TENANT_ID`, etc.)
  2. `scope.graph.tenant_id`, `scope.graph.client_id`, `scope.graph.client_secret`
  3. `scope.sources.find(s => s.type === 'graph')` fields
- Report which source was checked and where the gap is.

### 4. Complete `.env.example`

In `packages/ops-kit/src/commands/init-repo.ts`:
- Add `GRAPH_ACCESS_TOKEN=` to the generated `.env.example`.
- Add `NARADA_OPENAI_API_KEY=` to the generated `.env.example`.

### 5. Daemon Config Detection

In `packages/layers/daemon/src/index.ts`:
- If `./config.json` is missing, try `./config/config.json` before failing.
- Log which config path was used on startup.

### 6. `narada doctor` Command

Create `packages/layers/cli/src/commands/doctor.ts`:
- Check daemon process is running (PID file or process lookup).
- Check `.health.json` exists and `status === 'healthy'`.
- Check last sync was within staleness threshold.
- Check no `failed_retryable` / `failed_terminal` work items.
- Report concise pass/fail with actionable next steps.
- Exit code 0 if healthy, 1 if degraded.

## Non-Goals

- Do not implement secure-storage CLI helpers (deferred).
- Do not implement `narada setup-systemd` automation (deferred).
- Do not change the internal config schema shape.

## Execution Notes

### Unify Init Commands
- `config.ts`: Removed `mailbox_id` from `DEFAULT_CONFIG`. Added deprecation warning at start of `configCommand` — logs via `logger.info` and prints via formatter (`warning` level). The command still writes the config for backward compatibility but clearly directs users to `narada init-repo <path>`.
- `config-interactive.ts`: Changed default output path from `./config.json` to `./config/config.json`. Added `mkdir(dirname(outputPath), { recursive: true })` before `writeFile` so the `./config/` directory is created automatically.

### Extend `want-mailbox` CLI
- Added `--graph-user-id`, `--folders`, and `--data-root-dir` options to both `packages/layers/cli/src/main.ts` and `packages/ops-kit/src/main.ts`.
- The underlying `wantMailbox` function in ops-kit already supported these options; the change wires them through the CLI surface.

### Harden Preflight
- Rewrote Graph credentials check in `packages/ops-kit/src/readiness/collect.ts` to inspect three sources in order: env vars, `scope.graph` (legacy), and `graph` source in `sources[]`.
- Reports which sources were checked and which specific fields (`tenant_id`, `client_id`, `client_secret`) are missing.

### Complete `.env.example`
- Added `GRAPH_ACCESS_TOKEN=` and `NARADA_OPENAI_API_KEY=` to the `.env.example` generated by `init-repo.ts`.

### Daemon Config Detection
- Added fallback logic in `packages/layers/daemon/src/index.ts`: if `./config.json` is missing, tries `./config/config.json`. Logs the resolved path unconditionally on startup.

### `narada doctor` Command
- Created `packages/layers/cli/src/commands/doctor.ts` with four checks per scope:
  1. **daemon-process**: PID file existence + process liveness
  2. **health-file**: `.health.json` status === 'healthy'
  3. **sync-freshness**: last sync within configurable threshold (default 60 min)
  4. **work-queue**: `failed_retryable` / `failed_terminal` counts from coordinator DB
- Exit code 0 if all checks pass, 1 if any check is warn/fail.
- Registered under CLI `narada doctor` with `--stale-threshold-minutes` option.

### Tests
- `packages/layers/cli/test/commands/doctor.test.ts`: 4 tests covering degraded (no daemon), healthy (all green), degraded (failed work items), and invalid config.
- `packages/layers/cli/test/commands/config.test.ts`: Updated to assert `mailbox_id` is undefined and deprecation message is logged.
- `packages/ops-kit/test/unit/ops-kit.test.ts`: Added tests for `want-mailbox` new options and `.env.example` contents.
