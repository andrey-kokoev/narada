# Task 288 Implementation Plan: Autonomous Send as Approved-Draft Execution

## Summary

Split the current `SendReplyWorker` monolith into two distinct durable paths:
1. **Draft creation path** (`SendReplyWorker`): creates/reuses the Graph draft, stops at `draft_ready` for `send_reply` (was auto-sending).
2. **Send execution path** (`SendExecutionWorker`): NEW — picks up `approved_for_send` commands, performs the actual Graph send, transitions to `submitted`.
3. **Promotion surface**: NEW operator action `approve_draft_for_send` transitions `draft_ready` → `approved_for_send` with audit.

The `draft_reply` path remains unchanged: `draft_ready` → `confirmed`.

### Workers
- `packages/layers/control-plane/src/outbound/send-reply-worker.ts`
  - For `send_reply`: stop at `draft_ready` after draft creation/verification (remove the auto-send logic)
  - For `draft_reply`: unchanged (still `draft_ready` → `confirmed`)
  - Remove `sendDraft()` private method and policy-check logic (moves to new worker)
  - Update `processNext` candidate statuses for `send_reply`: still scans `pending`, `draft_creating`, `draft_ready`

- **NEW** `packages/layers/control-plane/src/outbound/send-execution-worker.ts`
  - Scans `send_reply` commands with status `approved_for_send`
  - Re-verifies managed draft (content hash match)
  - Runs policy gate (participant resolver)
  - Transitions `approved_for_send` → `sending`
  - Calls `draftClient.sendDraft()`
  - On success: transitions `sending` → `submitted`
  - On retryable failure: transitions `sending` → `retry_wait`
  - On auth/non-retryable failure: transitions `sending` → `failed_terminal`
  - Idempotent: if already submitted, no-op

## Invariants at Risk

| Invariant | Risk | Mitigation |
|---|---|---|
| **Outbound invariant 32**: Draft-First Delivery | Low — we still create draft first; send only happens after draft exists | Keep draft creation in `SendReplyWorker`; `SendExecutionWorker` verifies draft exists before send |
| **Outbound invariant 33**: Two-Stage Completion (submitted vs confirmed) | Low — task explicitly preserves this | `SendExecutionWorker` only sets `submitted`; reconciler remains sole path to `confirmed` |
| **Outbound invariant 34**: No External Draft Mutation | Low — same verification logic moves to new worker | `SendExecutionWorker` re-runs content-hash verification before send |
| **Control Plane invariant 11**: `OutboundHandoff` owns command creation | None — no change to command creation path | Foreman + `OutboundHandoff` unchanged |
| **Control Plane invariant 12**: Outbound workers own mutation | Low — new worker is still an outbound worker | `SendExecutionWorker` lives in `outbound/` and is registered under `executor_family: 'outbound'` |
| **State machine validity** | Medium — new state must have valid transitions | Add `approved_for_send` to `VALID_TRANSITIONS` with exactly `sending`, `cancelled`, `superseded` |
| **Idempotency of send** | Medium — crash after send but before `submitted` update | Leave in `sending` for reconciler; same as current behavior |
| **Eligibility uniqueness** | Low — `isVersionEligible` changes | Only one version can be latest + non-superseded + `approved_for_send` at a time |

## Dependency Assumptions

1. `GraphDraftClient.sendDraft()` API is stable and idempotent-enough at Graph level (same as current assumption).
2. `ParticipantResolver` and `draftClient` interfaces are unchanged; new worker consumes same deps.
3. `WorkerRegistry.drainWorker()` iterates all registered IDs; adding a new ID is sufficient.
4. SQLite schema migrations via `alter table` are acceptable for the `approved_at` column.
5. The existing `SendReplyWorker` tests can be updated in-place; no cross-package consumers depend on auto-send behavior.
