# Task 291: Live Graph Proof Saturation

## Chapter

Mailbox Saturation

## Context

Narada now has a coherent mailbox pipeline, including autonomous send as an approved-draft execution path. What is still missing is a durable, repeatable acceptance proof for the full live Microsoft Graph loop: draft creation, approval/send submission, and reconciliation after real send.

The public repo must not absorb private mailbox content or live credentials. The proof shape must therefore be split cleanly between public acceptance contract and private operational evidence.

## Goal

Define and execute the canonical live-mailbox proof shape for Narada’s first mailbox vertical, with clear public acceptance artifacts and private-ops execution boundaries.

## Required Work

### 1. Define the live proof contract

Specify the exact proof stages:

- inbound thread selected
- evaluation produced
- outbound draft created
- draft approved for send
- send submitted
- reconciliation confirms sent result

For each stage, define:

- durable rows or files that must exist
- the canonical command/query used to inspect them
- what belongs in the public repo vs the ops repo

### 2. Harden the private/public boundary

Document the split between:

- public: contract, safe fixtures, expected evidence shape
- private ops repo: real mailbox IDs, live transcripts, private acceptance logs

No private message bodies or credentials may be required inside the public repository.

### 3. Add an operator-proof runbook

Write the shortest coherent runbook for a real operator to execute the proof from:

- sync
- inspect
- approve
- send
- verify confirmation

### 4. Verify against the current surfaces

Confirm the proof contract matches the actual CLI, daemon UI, and durable-state surfaces that now exist.

## Non-Goals

- Do not add new architectural abstractions.
- Do not store private mailbox data in the public repo.
- Do not broaden this into generalized production monitoring.

## Acceptance Criteria

- [x] A canonical live-proof contract exists for the full draft-to-confirmed path.
- [x] The public/private evidence split is explicit and coherent.
- [x] A focused operator runbook exists for executing the proof.
- [x] The proof contract references current real surfaces, not stale commands.
